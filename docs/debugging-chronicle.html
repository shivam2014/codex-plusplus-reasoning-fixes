How 5 bugs — one a blank-screen syntax error — were found and fixed

The codex-plusplus-reasoning-fixes tweak intercepts Codex's renderer bundles and patches JavaScript at load time. In the span of one evening, 5 independent bugs were identified and fixed across the fix>codex_update_26.506.20924 branch. Two of them needed multiple attempts. One produced a completely blank conversation view.

TL;DR: A chain of 5 commits fixed a blank-screen syntax error (declaration inside a declaration), 3 bundle-target drifts where patches silently stopped working, a toggle that did nothing, a stale-closure cleanup failure, and 3 wiring gaps. Two of the fixes introduced follow-on bugs that the next commit caught — a vivid example of how source patching amplifies small mistakes.

Overview

Before diving into each bug, it helps to understand the architecture — because every bug traced back to a mismatch between what the patcher expects and what Codex actually serves.

Architecture

Key insight: The patcher has two independent registration systems. PATCHES defines text transformations. SETTING_FEATURES maps settings to which patches should activate. FEATURES (in index.js) handles runtime DOM effects that don't need source patching. When these three systems get out of sync, bugs emerge.

Bug 3: The cleanup that silently failed

Stale closure in _collapseCleanup after idempotency guard
closure
commit

afde18c

observation

After hot-reload, stopping the plugin left the collapse-all button and CSS style element orphaned in the DOM.

root cause

Fix 2 added an idempotency guard (if (document.getElementById("rft-btn")) return;). When this fires, local variables btn and styleEl are never assigned. The cleanup closure captured those null variables and silently did nothing.

fix

Look up DOM elements at cleanup time instead of capturing in a closure.

+ _collapseCleanup = function() { + var b = document.getElementById("rft-btn"); + if (b && b.parentNode) b.parentNode.removeChild(b); + };

Bug 5: The blank-screen syntax error

5
fix-assistant-order produced invalid JS — SyntaxError → blank view
CRITICAL
commit

99f25ce

symptom

Blank conversation view. The chat rendered as an empty white page. No errors in the console beyond a generic SyntaxError.

root cause

The replacement started with let O=null; but the surrounding code already had let T=...,E=...,. This produced a let inside a comma-separated let declaration — a SyntaxError in strict mode. The bundle failed to parse, React never mounted.

This was a direct side-effect of Fix 1c: adding let to match the patched regex broke the surrounding context.

fix
- replacement: "let O=null;for(...){...}let k=..." + replacement: "O=null;for(...){...}let k=..."
tradeoff

The patched regex no longer matches (it still expects let), so inspectRule() returns "unsupported". But the patch works correctly. Priority: renderer correctness over status accuracy.

Retrospective

These 5 bugs share two recurring themes: registering in one system but not another, and assuming context (surrounding code, bundle location) that silently changes.

💡
Pattern detection: Every bug except Bug 3 (closure) was a registration or mapping gap. Bug 5 was the only bug that produced a visible crash — and it was the hardest to detect because it required understanding the surrounding code context where the replacement lands.

5 lessons for the next patch

🔗

"let" inside "let"

A replacement that's correct in isolation may break the surrounding declaration context. Always read the ~20 characters before and after the injection point.

🔒

Idempotency breaks closures

Adding an early-return guard silently invalidates closures that capture variables initialized below. Always check cleanup/unmount paths.

🔁

Multi-occurrence needs g

If a pattern can appear N times, use the g flag and >= 1 not === 1. Otherwise the patch is silently skipped.

🗑

Remove dead toggle UI

When a refactor removes a patch, also remove its UI toggle. Rendered-but-inert toggles erode user trust.

🧪

Validate replacements as JS

Run every patch's replacement as .mjs through a syntax check. A 2-line script would have caught Bug 5 before deployment.

🗺

Version-pin bundle targets

Store the Codex version alongside bundle mappings and warn on drift. The current system says "unsupported" which doesn't guide repair.
