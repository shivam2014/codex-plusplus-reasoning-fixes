Auto-Healing Engine

Each of the 15 patches carries a skeleton: a loose regex with (\w+) capture groups for every JavaScript identifier. When the exact-match regex fails after a Codex update, the engine falls through to the skeleton.

Inspect → Heal → Verify

1. inspectRule() → unpatched=0 patched=0 → "unsupported"
2. autoHeal() → load skeleton match
→ /if\((\w+)\.type===`reasoning`\)\{(\w+)&&\2\.push(\1);continue\}/
3. matchSkeleton() → captures: {1: "t", 2: "i"}
4. regenerate replacement → inserts captured names into replacement template
5. verify → run skeleton.verify against patched output → ✓
6. cache → save to api.storage per Codex version → survives restarts
Concrete example: fix-assistant-order

When Codex updated from v26.519.22136 to v26.519.41501, the fix-assistant-order patch's exact regex stopped matching because minified variable names changed:

Old variables: D, E, O, Xe, k, g, T
New variables: w, C, T, me, E, h, S

The skeleton's (\w+) groups captured the new names and the replacement function regenerated the patched code using them. The patch worked on the first reload — no manual intervention.

Protocol Interception

v1.3 uses a dual approach to intercept app:// bundle requests. This was necessary because the Computer Use tweak (co.bennett.computer-use) registers protocol.interceptBufferProtocol("app", handler), which replaces the entire handler and bypasses protocol.handle wrapping.

Approach A — Patch protocol.handle
→ Wraps future protocol.handle("app", handler) registrations
→ Works when no other tweak uses interceptBufferProtocol

Approach B — Register protocol.interceptBufferProtocol("app", handler)
→ Calls uninterceptProtocol("app") first to remove any existing handler
→ Reads files directly from the ASAR bundle
→ Applies Statsig gate patches (replaces CU tweak's Statsig patching)
→ Applies all 15 reasoning-fixes patches with auto-healing
→ Retries with staggered delays [0,50,150,500,1500,3000,6000,12000]ms
Why both approaches?

Before v1.3, only Approach A existed. When the Computer Use tweak's interceptBufferProtocol succeeded (which started happening in Codex v26.519.41501), it replaced the app:// handler entirely and the reasoning-fixes patcher never saw any requests. The log showed "source patcher v2 ready" but no bundles were ever patched.

Approach B fights back: it unregisters any existing intercept, registers its own handler, and keeps retrying. Since it also handles Statsig gate patching, the CU tweak's functionality is preserved — just running through RF's handler instead.
